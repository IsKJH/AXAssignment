# ① 앱 (슬라이드 1~4, 약 4분)

## 슬라이드 1 — 타이틀 (30초)

안녕하세요. 과제로 가계부 앱을 만들었습니다.
앱이 어떤 모습인지, 어떤 생각으로 설계했는지 먼저 말씀드리고,
이어서 이 앱을 어떻게 만들었는지 — 개발하면서 뭐가 막혔고 어떻게
풀었는지를 말씀드린 뒤, 마지막에 실제 폰으로 시연하겠습니다.

## 슬라이드 2 — 어떤 앱인가 (1분 30초)

핵심 기능은 네 가지입니다.

- **나만의 주기**: 1일~말일이 아니라 **내가 정한 날짜, 예를 들어 월급날
  기준**으로 한 달을 봅니다. 모든 화면이 이 주기 위에서 돌아갑니다.
- **정기 거래**: 월급이나 월세처럼 매달 반복되는 건 한 번 등록하면
  매달 알아서 들어갑니다.
- **9개 화면 전부 구현**: 홈, 거래 추가/상세, 카테고리, 통계,
  6개월 내역, 설정, 가이드 투어까지.
- 그리고 전 화면이 **Figma 디자인과 대조됐고, 실제 폰에서 테스트까지
  끝난 상태**로 GitHub에 올라가 있습니다.

## 슬라이드 3 — 설계 (1분 30초)

설계에서 챙긴 건 세 가지입니다.

첫째, **MVVM + Repository 구조.** 화면, 로직, 데이터를 분리했고
데이터는 DB에서 ViewModel을 거쳐 화면으로 **한 방향으로만** 흐릅니다.
그래서 버그가 숨을 곳이 없습니다. 색과 글꼴도 전부 디자인 토큰으로
관리해서 디자인이 어긋날 수 없는 구조입니다.

둘째, **제일 공들인 기능은 정기 거래**입니다. 한 번 등록하면 매달
자동으로 들어가고, 앱을 켤 때마다 알아서 연장됩니다. 수정이나 삭제할
때는 "이번 달만 / 이후 전체 / 전체" 중에 고를 수 있습니다.

셋째, **명세에 없던 것들.** 예를 들어 수입의 카테고리 — 명세의 기본
카테고리 7개가 전부 지출용이더라고요. 그래서 수입은 카테고리 없이 가기로
정하고, 그 근거를 기록해뒀습니다. 명세의 빈칸은 추측이 아니라
**근거를 적은 결정**으로 채웠습니다.

## 슬라이드 4 — 디테일 + 일화 (45초)

디테일도 챙겼습니다. 통계 도넛을 누르고 문지르면 카테고리가 실시간으로
바뀌고, 숫자가 차오르는 애니메이션, 리스트가 차례로 등장하는 효과,
가로모드까지. 이건 마지막 시연에서 직접 보여드리겠습니다.

재미있는 게 하나 있는데 — **오늘 아침, 발표일에 맞춰 주기가 자동으로
넘어갔고, 월급과 월세가 새 주기에 미리 등록돼 있었습니다.**
만든 기능이 발표 당일 실전에서 작동한 거죠.

# ② 어떻게 만들었나 (슬라이드 5~10, 약 5분 30초)

## 슬라이드 5 — 전환 (40초)

여기까지가 앱이고, 이제 만든 방식입니다.
사실 — **이 앱의 코드는 전부 AI가 작성했습니다.** 도구는 Claude Code를
썼고, 저는 코드를 친 게 아니라 AI가 일할 수 있는 **환경을 설계**하고,
개발하다 막히는 곳마다 **직접 도구를 만들어서** 뚫었습니다.
지금부터 그 과정입니다.

## 슬라이드 6 — AI에게 무엇을 쥐여줬나 (1분)

AI에게 세 가지를 쥐여줬습니다.

- **MCP 서버** — AI에게 손과 눈을 달아주는 도구 연결입니다.
  figma로 디자인을 그림이 아니라 수치 데이터로 읽게 하고,
  mobile-mcp로 제 갤럭시 폰을 직접 조작하게 했습니다.
- **스킬과 에이전트** — "화면 만들어줘" 한마디가 들어오면 항상 같은
  3단계 파이프라인이 돌도록 절차를 고정했고, 단계마다 그 일만 하는
  전담 에이전트를 뒀습니다.
- **설정과 훅** — 위험한 명령은 무조건 차단, 빌드 같은 일상 작업은
  승인 없이 자동, 설정은 전부 git으로 백업해서 언제든 롤백.

핵심 발상은 — AI에게 일을 시키는 게 아니라,
**AI가 일할 수 있는 공장을 짓는다**는 겁니다.

## 슬라이드 7 — 3단계 파이프라인 (1분 30초)

"홈 화면 만들어줘" 한마디에 일어나는 일입니다.

**1단계, 읽기(figma-reader).** Figma에서 요구사항과 디자인 수치를 읽어서
명세서를 만듭니다. 이 명세서엔 "무엇이 화면에 보이면 합격인지"까지
적힙니다. 참고로 처음엔 AI가 Figma 안의 스케치 그림을 진짜 디자인으로
착각했어요. 그래서 "글로 적힌 요구사항만 믿고, 디자인은 디자인 프레임에서
실측해라"라는 읽기 규칙부터 만들어야 했습니다.

**2단계, 만들기(compose-builder).** 명세서대로 Compose 코드를 작성합니다.
컴파일을 통과하지 못하면 다음 단계로 못 갑니다.

**3단계, 검사(device-qa).** 실제 폰에 설치해서 화면을 명세와 대조하고
Pass/Fail 리포트를 만듭니다. Fail이면 고치고 다시 검사.
여기까지 **사람 개입이 없습니다.** 사람이 하는 건 링크 주기,
질문에 답하기, 최종 확인 — **결정은 사람, 노동은 시스템.**

## 슬라이드 8 — 물론 한 번에 되진 않았습니다 (1분 30초, 핵심)

이 과정이 처음부터 매끄러웠던 건 아닙니다. 세 번 막혔습니다.

**첫째, 폰 조작이 너무 느렸습니다.** mobile-mcp는 매번 화면을 찍고,
그림에서 버튼 위치를 추측해서 누르는 방식이라 — 거래 1건 입력하는 데
**107초**가 걸렸습니다. 그래서 두 가지를 했습니다. 자주 쓰는 버튼은
좌표를 실측해서 기록해두고, **화면의 UI 트리에서 글자로 버튼을 찾아
누르는 전용 드라이버(ui.py)를 직접 만들었습니다.**
결과는 **13초, 8배 빨라졌습니다.**

**둘째, 화면 확인이 너무 비쌌습니다.** AI가 화면을 한 번 확인할 때마다
스크린샷으로 토큰이 2,000씩 나갔는데, "이 글자가 화면에 있나?"만
확인하는 텍스트 추출 명령을 추가해서 **90% 줄였습니다.**

**셋째, 빌드 기다림.** 검사할 때마다 빌드 대기가 쌓여서,
gradle 빌드 캐시로 5초를 1초로 줄였습니다.

패턴은 하나입니다 — **느린 곳을 재고, 도구를 만들고, 다시 잰다.**

## 슬라이드 9 — 스스로 좋아지는 시스템 (1분 30초)

제일 말씀드리고 싶은 부분입니다. 방금 같은 개선이 한 번으로 끝나지 않고
**시스템에 쌓이게** 만들었습니다.

- 도구가 실패하면 → 훅이 백로그에 자동 기록하고, 다음 세션의 AI가
  원인을 제거합니다. 실패가 곧 개선 과제가 되는 구조입니다.
- 작업을 시작할 때마다 → 시스템 이상을 자동 점검해서 경보합니다.
- 매일 밤 → AI가 혼자 신기술과 새 도구를 조사해두고, 아침에 제가 검토합니다.
- 3일마다 → 설정과 문서 전체를 스스로 감사합니다.

실화 하나 — 발표 전날 "스스로 검증해봐"라고 한마디 했더니, AI가
**에이전트 3개를 띄워 자기 자신을 교차 검증**해서 숨은 버그 5개를 찾아
고쳤습니다. 물론 안전장치도 있습니다. 위험한 명령은 무조건 차단되고,
밤에는 조사만 하고 수정은 사람이 있을 때만, 설정은 전부 백업돼 있습니다.

## 슬라이드 10 — 다음 앱은 더 빠릅니다 (45초)

이 프로젝트에서 검증된 체계 전체 — 파이프라인, 기기 드라이버, 자동화
규칙 — 는 전역 스킬로 추출해뒀습니다. 다음에 앱을 만들 땐
"새 앱 만들자" 한마디면 같은 체계가 깔리고, 기획만 주면 명세화부터
구현, 검증까지 같은 흐름이 돕니다. 그리고 쓰면 쓸수록 실패 기록과
신기술 조사가 쌓여서 체계 자체가 자랍니다.

**제가 만든 건 가계부 앱 하나가 아니라, 다음 앱부터 더 빨라지는
개발 체계입니다.**

# 라이브 시연 (슬라이드 11, 약 2분 30초)

이제 실제 폰으로 보여드리겠습니다. *(scrcpy 미러링)*

**시연 동선:**
1. **홈** — 주기별 수입/지출/진행률 확인 → **좌우 스와이프**로 이전 주기 이동
   (이전 주기에 풍성한 데이터)
2. **거래 추가** — FAB → 금액·카테고리·날짜 입력 → 저장 (토스트·리스트 애니메이션)
3. **거래 상세** — 방금 저장한 거래 탭 → 내용 확인 → 삭제 (햅틱 확인 다이얼로그)
4. **통계** — 도넛 **누르고 문지르기** (카테고리 실시간 전환 + 햅틱) →
   막대 탭 → 6개월 내역
5. **설정** — 주기 시작일 화면 → 가이드 투어 한두 페이지
6. (여유 있으면) **가로모드** 회전 한 번

마치겠습니다. 감사합니다.
