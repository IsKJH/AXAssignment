01

실험 개요

동일한 요구사항(Open-Meteo API 연동)을 주었을 때 각 AI가 어떤 HTTP 라이브러리와 아키텍처를 자율적으로 선택하는지를 관찰했습니다. 프롬프트는 "목 데이터를 Open-Meteo로 교체하라"는 것 외에 라이브러리나 설계 방식을 지정하지 않았습니다.

비교 초점
단일 목표(API 교체) 앞에서 AI가 내린 선택 — HTTP 라이브러리(외부 vs 내장), 네트워크 레이어 구조(분리 vs 통합) — 가 이후 단계의 테스트 작성 난이도와 코드량에 어떻게 이어지는지를 추적합니다. 실기기에서 세 앱 모두 실데이터 수신이 확인됐습니다.
핵심 해석
Claude와 Codex는 Android에 기본 내장된 표준 라이브러리(stdlib)만 사용해 외부 의존성 없이 구현했습니다. Gemini는 업계 표준인 Retrofit 스택을 선택해 코드 선언성을 높인 대신 라이브러리 3개를 추가했습니다. Claude는 여기에 더해 네트워크 로직을 WeatherRepository라는 독립 클래스로 분리했고, 이 설계가 V5 단계의 테스트 용이성을 결정했습니다. Gemini는 V3 스마트 기능(온도 배경·인사이트 카드)을 V4에서도 그대로 유지했습니다.
주목할 차이
세 앱이 같은 Open-Meteo API를 호출했음에도 풍속(Codex 1.6 vs Claude·Gemini 5.7 m/s)과 대기질 표현(Codex 수치 64 vs Gemini "미세먼지 좋음")이 달랐습니다. API 응답의 어떤 필드를 어떻게 매핑했느냐에 따른 차이로, 동일한 데이터도 구현에 따라 사용자에게 다른 값으로 보일 수 있음을 보여줍니다.

라이브러리 선택의 트레이드오프: stdlib(표준 라이브러리) 방식은 외부 의존성이 없어 빌드 설정이 단순하고 라이브러리 버전 충돌 위험이 없습니다. Retrofit 방식은 초기 설정 비용이 있지만, API 엔드포인트를 인터페이스로 선언해 코드 가독성이 높고 대형 프로젝트에서 팀이 분업하기 쉽습니다. 이 실험의 단일 앱 규모에서는 두 방향 모두 빌드에 성공했으며, 어느 쪽이 우위라고 판단하기보다 상황에 따른 선택 차이로 관찰하는 것이 적절합니다.
