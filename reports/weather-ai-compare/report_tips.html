01

지시서 파일 — AI에게 프로젝트 규칙 미리 알려주기

모든 AI 도구에는 "이 프로젝트에서 어떻게 행동해야 하는지"를 기록해두는 파일이 있습니다. CLAUDE.md는 Claude Code가 프로젝트 시작 시 자동으로 읽는 규칙 파일로, 아키텍처 지침·금지 패턴·빌드 명령 등을 한 번만 적어두면 매 대화에서 반복하지 않아도 됩니다. 이 파일이 없으면 AI는 매 대화마다 처음부터 맥락을 파악하고, 이미 알려진 실수를 반복합니다.

도구파일명위치공식 문서
Claude Code CLAUDE.md 프로젝트 루트 · ~/.claude/ (전역) 공식 문서 →
OpenAI Codex AGENTS.md 프로젝트 루트 · ~/.codex/ (전역) 공식 문서 →
Gemini CLI GEMINI.md 프로젝트 루트 · ~/.gemini/ (전역) 공식 문서 →

Android 프로젝트용 최소 CLAUDE.md 예시

# WeatherNow Android ## 빌드 명령 - `./gradlew assembleDebug` # gradlew(그레이들 래퍼 — 빌드 도구 실행 스크립트) · assembleDebug = 디버그 APK 빌드 - `./gradlew test` # 단위 테스트 ## 금지 규칙 - CoroutineWorker에서 runBlocking 사용 금지 - 기존 단위 테스트 수정 금지 (새 테스트 추가는 허용) - V5 이후 테스트가 통과하지 않으면 커밋하지 말 것 ## 아키텍처 - data/ → Repository 구현체, Room DB - domain/ → Entity, Repository 인터페이스 (DIP 준수) - ui/ → Composable, ViewModel
💡
Karpathy의 4규칙 (CLAUDE.md 핵심): ① 모호하면 짜기 전에 물어봐라 ② 요청한 것만 짜라 (불필요한 추상화 금지) ③ 관련 없는 코드 건드리지 마라 ④ 막연한 요청은 검증 가능한 기준으로 변환해라 ("버그 고쳐" → "이 테스트를 통과시켜"). 파일은 200줄 이하로 유지하는 것이 원칙입니다.
02

Sub-agents — 메인 AI가 전문 AI팀에 작업을 분배

Claude Code 최신 공식 문서 확인됨

하나의 큰 작업을 여러 전문 에이전트에게 나눠서 처리합니다. 각 서브에이전트는 독립된 컨텍스트 윈도우(AI가 한 번에 처리할 수 있는 입력 길이 한계)를 가지므로, 메인 대화에 불필요한 내용이 쌓이지 않고 토큰 비용을 5~10배 절감할 수 있습니다.

패턴구조언제 씀
Fan-out 메인 → 서브A + 서브B + 서브C (병렬) 여러 파일을 동시에 분석·수정할 때
Pipeline 서브A → 서브B → 서브C (순차) 분석 → 구현 → 테스트 단계가 있을 때
Producer-Reviewer 서브A(생성) → 서브B(검증) 코드 생성 후 품질 검증이 필요할 때

Android 개발 활용 예시

# 메인 에이전트에게 지시 "WeatherViewModel을 리팩토링해줘. 서브에이전트 A는 기존 테스트 분석을, 서브에이전트 B는 ViewModel 구현을 담당해. A가 끝난 후 B가 테스트를 깨지 않는 방향으로 구현해." # 결과: 메인 컨텍스트에 테스트 전체 내용이 안 쌓임 # 각 서브에이전트가 자기 역할만 처리하고 결과만 보고
📋
이 실험과의 연결: 이번 실험에서 Claude·Codex·Gemini 각각이 순차적으로 같은 작업을 했습니다. Sub-agents를 쓰면 세 AI가 병렬로 같은 작업을 동시에 처리하고 결과를 비교하는 것이 가능합니다. 이 실험 자체를 Sub-agents로 자동화할 수 있다는 의미입니다.
03

Hooks — AI가 뭔가 할 때마다 자동으로 실행되는 규칙

Claude Code 공식 기능

AI가 파일을 쓰거나, 터미널 명령을 실행하거나, 커밋하기 전에 자동으로 쉘 명령을 실행합니다. 개발자가 확인하지 않아도 항상 실행되는 규칙입니다.

라이프사이클 이벤트

PreToolUse
도구 실행 전
위험한 명령 차단. exit 2로 종료하면 Claude가 해당 동작을 중단합니다.
PostToolUse
도구 실행 후
파일 수정 후 자동 포맷팅·린팅 실행. 코드 품질을 사람이 확인하지 않아도 유지합니다.
PostWrite
파일 저장 후
Kotlin 파일 저장 시 ktlint 자동 실행. 커밋 전에 린트 에러가 없음을 보장합니다.

Android 개발 Hooks 예시

// .claude/settings.json { "hooks": { "PostWrite": [ { "matcher": "**/*.kt", "command": "cd $CLAUDE_PROJECT_DIR && ./gradlew ktlint -q" } ], "PreToolUse": [ { "matcher": "Bash", "command": "echo '$CLAUDE_TOOL_INPUT' | grep -q 'rm -rf' && exit 2 || exit 0" } ] } }
🔧
이 실험과의 연결: V6에서 Gemini가 버그를 수정했지만 JAVA_HOME 환경 오류로 빌드 검증을 완료하지 못했습니다. PostWrite Hook에 ./gradlew assembleDebug를 걸어두면 파일이 저장될 때마다 자동으로 빌드를 확인합니다. 검증 누락이 구조적으로 불가능해집니다.
04

MCP — AI가 외부 도구를 직접 읽고 쓰는 표준 프로토콜

MCP(Model Context Protocol — AI 도구와 외부 서비스 연동 표준)는 AI가 GitHub, Firebase, Jira, Slack, 데이터베이스 등 외부 도구와 직접 연결되는 표준 방식입니다. 정보를 복사·붙여넣기하지 않아도 AI가 직접 도구에서 읽고 씁니다.

MCP 서버기능Android 개발 활용
android-mcp-server ADB(안드로이드 기기 연결 도구) 제어, 스크린샷, UI 트리 분석 AI가 실기기에서 앱 직접 조작 및 버그 확인
github PR 생성/조회, 이슈 읽기, 코드 검색 "이 이슈 분석하고 PR 만들어줘" 한 번에
filesystem 파일 시스템 직접 접근 프로젝트 전체를 컨텍스트로 활용
firebase Firestore 쿼리, Auth 관리 프로덕션 데이터를 직접 분석·디버깅
# Claude Code에 Android MCP 추가 (터미널) claude mcp add android-mcp-server # 그러면 이런 프롬프트가 가능해짐: "앱 설치하고 서울 탭 눌러서 날씨가 뜨는지 스크린샷 찍어서 확인해줘" # V8 실험에서 ADB로 WorkManager(백그라운드 작업 스케줄러)를 수동 트리거했던 것을 AI가 직접 함
📱
이 실험과의 연결: V8에서 각 AI가 직접 ADB 명령으로 WorkManager를 트리거하고 스크린샷을 찍었습니다. MCP 설정 시 이 과정 전체를 AI가 자율적으로 수행합니다. "알림이 실제로 왔는지 확인해줘"라는 한 문장으로 V8 실기기 검증 전체를 자동화할 수 있습니다.
05

GitHub Actions 연동 — PR마다 AI 자동 리뷰

Claude Code 공식 기능

PR을 올릴 때마다 Claude가 자동으로 코드를 리뷰하고, @claude를 멘션하면 즉시 수정안을 제시합니다. CI/CD 파이프라인에 AI 리뷰어를 추가하는 것입니다.

# 설치 (Claude Code CLI에서) claude /install-github-app # PR 코멘트에서 사용 @claude 이 ViewModel에서 메모리 누수 가능성 있는 부분 확인해줘 @claude suspend fun 대신 runBlocking 쓴 곳 찾아서 수정해줘
자동 코드 리뷰
PR 열 때마다 분석
새 PR이 올라오면 Claude가 자동으로 잠재적 문제점, 개선 제안을 코멘트로 남깁니다.
@claude 멘션
즉석 수정 요청
PR 코멘트에서 @claude를 멘션하면 해당 코드를 직접 수정하는 PR을 만들어줍니다.
보안 감사
취약점 자동 탐지
매 커밋마다 보안 취약점을 체크하고 경고를 남깁니다. 코드 리뷰어 역할을 AI가 합니다.

07

AI와 대화하면서 시작하기 — 설정을 처음 만드는 방법

CLAUDE.md도, 테스트 케이스도, AI에게 초안을 부탁하면 됩니다. 단 "처음부터 완벽하게"를 목표로 하지 말고, 실제로 AI가 실수할 때마다 규칙을 한 줄씩 추가하는 방식이 훨씬 효과적입니다.

CLAUDE.md 초안 — 실제 대화 흐름

# 1단계: AI에게 현재 프로젝트 설명 → 초안 부탁 (5분) "지금 Android Kotlin MVVM 프로젝트야. Claude Code에 쓸 CLAUDE.md 초안 만들어줘. 빌드는 ./gradlew assembleDebug, 기존 테스트 파일 수정은 금지야." # 2단계: 실제로 써보기 → AI가 실수하면 규칙 한 줄 추가 "AI가 CoroutineWorker에서 runBlocking을 써서 실기기 크래시가 났어. 이 패턴 금지 규칙 추가해줘." # 3단계: 반복 → 3개월 후엔 팀만의 맞춤 규칙북이 됩니다
처음엔 5줄도 충분
완벽함보다 시작이 중요
빈 파일보다 5줄짜리 CLAUDE.md가 훨씬 낫습니다. AI와 함께 일하면서 "이 실수는 규칙으로 막자"는 순간마다 한 줄씩 추가하세요.
실제 실수에서 나온 규칙이 최강
추상 규칙 vs 경험 규칙
"잘 짜줘" 같은 추상적 규칙보다 이번 실험의 "CoroutineWorker에서 runBlocking 금지"처럼 실제 사고에서 나온 규칙이 훨씬 효과적입니다.
팀 공유 자산
개인 → 팀 규칙으로
한 팀원이 겪은 AI 실수를 CLAUDE.md에 추가하면, 다른 팀원은 같은 실수를 처음부터 겪지 않습니다. 팀의 집단 지성이 됩니다.
08

테스트 먼저, 코드 나중 — 검증 가능한 목표를 주는 방법

AI에게 막연한 요청을 하면 막연한 결과가 나옵니다. "이 테스트를 통과하는 X를 만들어줘" 형태로 바꾸면 AI가 스스로 검증하면서 작업하고, 결과가 맞는지 틀린지를 사람이 직접 확인하지 않아도 됩니다.

❌ "로그인 기능 만들어줘"
→
AI가 뭘 만들어도 맞는지 판단하기 어렵습니다
✅ "이메일 형식 검증 / 잘못된 비밀번호 에러 처리 / 성공 시 토큰 저장 — 이 3가지 테스트 케이스를 먼저 만들고, 그걸 통과하는 로그인 기능을 구현해줘"
→
AI가 스스로 테스트를 실행하며 검증. Hook이 있으면 저장할 때마다 자동 실행

주의: AI가 만든 테스트의 함정

나쁜 테스트 예시
뭘 반환해도 통과
assert result is not None
이런 검증은 버그가 있어도 통과됩니다. AI가 자신이 짠 코드를 통과시키기 쉽게 테스트를 쓰는 경향이 있습니다.
좋은 테스트 예시
실패 케이스를 정확히 잡음
assert error.message == "비밀번호가 틀렸습니다"
정확한 에러 메시지, 상태 코드, 반환값 형식을 검증해야 진짜 버그를 잡습니다.
최소 검토 포인트
테스트 받은 후 한 번만
AI에게 "이 테스트에서 X 조건이 실패하면 어떤 에러가 나와?"라고 물어보세요. 설명하지 못하면 테스트가 약한 겁니다.
📋
테스트 케이스도 AI와 대화로 만들면 됩니다. 단, AI가 테스트도 쓰고 코드도 쓰면 "AI의 셀프 합격"이 될 수 있습니다. "이 테스트가 진짜 실패 케이스를 잡을 수 있어?"라고 한 번 더 물어보는 습관이 AI 결과물의 신뢰도를 높입니다.
09

팀 적용 로드맵 — 단계별 자동화 수준 높이기

한 번에 모든 걸 설정하려 하지 마세요. 레벨별로 단계를 밟으면 2~3개월 안에 AI가 실수를 구조적으로 하기 어려운 환경이 만들어집니다.

레벨할 것효과목표 기간
Lv.1
기초 설정		 AI와 대화로 CLAUDE.md 초안 작성
실수할 때마다 규칙 한 줄 추가		 AI가 같은 실수를 반복하지 않음 1~2주
Lv.2
테스트 패턴		 "테스트 먼저 → 구현" 패턴 습관화 AI 결과를 믿지 않고 검증하는 구조 2~4주
Lv.3
자동화 기반		 PostWrite Hook으로 테스트 자동 실행 셋업 AI가 코드 수정 시 테스트 결과 자동 피드백 — 사람이 개입 안 해도 됨 3~5주
Lv.4
완전 자동화		 GitHub Actions + @claude PR 자동 리뷰
Sub-agents로 병렬 작업 분배		 AI가 AI를 검증하는 구조 — 사람은 최종 판단만 6~8주

4주 실전 루틴

1주차
환경 셋업
AI에게 현재 프로젝트 설명 → CLAUDE.md 초안 요청 → 실제 작업에 투입 → 문제 생기면 규칙 추가
2주차
테스트 먼저 패턴
작은 기능 하나를 "테스트 먼저 → 구현" 순서로 만들어보기. AI가 만든 테스트를 직접 읽고 의미 있는지 확인
3주차
자동화 추가
PostWrite Hook으로 파일 저장 시 테스트 자동 실행 셋업. 이제 AI가 코드를 수정하면 결과가 자동으로 피드백됩니다
4주차
회고 + 개선
AI가 가장 많이 실수한 패턴을 CLAUDE.md 규칙으로 추가. 팀원과 공유 → 다음 달 반복
🎯
자동화 수준의 기준: AI에게 일을 시키고 나서 직접 검증하지 않아도 되는 비율이 얼마나 되는가. Hook, 테스트, 자동 리뷰가 대신 검증해줄수록 높은 수준입니다. 목표는 "AI를 믿는 것"이 아니라 "믿지 않아도 되는 구조를 만드는 것"입니다.
