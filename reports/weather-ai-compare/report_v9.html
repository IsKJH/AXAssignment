실험 설계
Claude는 Codex·Gemini를, Codex는 Claude·Gemini를, Gemini는 Claude·Codex를 리뷰했습니다. 각 AI는 실제 파일을 직접 읽고 아키텍처·코루틴·Worker·테스트·DB 5개 차원으로 평가한 뒤, 다른 코드에서 배운 것을 자기 코드 개선 계획으로 전환했습니다.
핵심 질문
AI는 자기 코드의 부족한 점을 남의 코드에서 발견할 수 있는가? 발견했다면 그것을 "다음 버전에서 이렇게 바꾸겠다"는 구체적인 계획으로 전환할 수 있는가? V9는 코드를 새로 짜는 능력이 아니라 남의 코드를 이해하고, 비판하고, 자기 것으로 흡수하는 능력을 측정합니다.
특이사항
Gemini CLI는 초기 실행 시 워크스페이스 밖 파일 접근이 제한됐지만, 리포 루트에서 재실행 후 Claude·Codex 코드 파일을 직접 읽고 리뷰를 작성했습니다. FavoriteCityRepository, fetchJob?.cancel(), FakeOpenMeteoServer 등 실제 코드를 참조한 구체적인 리뷰가 완성됐습니다.

03

교차 평가 — 3개 AI가 동의한 것, 다르게 본 것

3개 AI 모두 동의한 관찰

🏆
Codex의 FakeOpenMeteoServer 테스트 전략 — 가장 높은 평가 수렴: FakeOpenMeteoServer는 실제 네트워크 요청 없이도 HTTP 통신을 흉내내는 테스트용 가짜 서버입니다. Claude는 이를 "가장 낮은 레이어에서 네트워크를 가로채 실제 스택에 가깝게 테스트하는 방식"으로, Gemini는 "네트워크 레이어의 결함을 조기에 발견할 수 있는 강력한 도구"로 평가했습니다. Codex 자신도 "다른 AI의 코드에서 1순위로 배우고 싶은 항목"으로 꼽았습니다. 3개 AI가 동일하게 Codex V5 테스트 전략을 최고 수준으로 평가한 유일한 항목입니다.
⚠️
Codex Worker 목 데이터 — 공통 비판: Claude는 "V4부터 실 API를 사용했음에도 V8에서 퇴보한 결정", Gemini는 "실용성 측면 아쉬움", Codex 자신도 "3순위 자가 개선 항목으로 실 API 전환"을 계획. 서로 다른 관점으로 접근했지만 결론이 일치.
✨
Gemini의 generateInsight() — 창의적 기능 인정: generateInsight()는 날씨 데이터를 받아 "기온이 낮고 습도가 높으니 우산을 챙기세요"처럼 사람이 바로 이해할 수 있는 자연어 해석 문구를 생성하는 함수입니다. Claude는 이를 "V3에서 도입해 V8까지 유일하게 유지된 스마트 기능"이라 평가했고, Codex는 "날씨 수치를 해석된 형태로 제공한다는 점이 경쟁 코드보다 우수"라고 인정했습니다. 두 AI 모두 Gemini가 V3 이후 이 기능을 꾸준히 유지한 것을 강점으로 봤습니다.

순환 학습 구조 발견 — "순환 학습 고리"란 무엇인가: 학습이 한 방향으로만 흐르지 않고 세 AI 사이에서 돌고 돈다는 뜻입니다. Claude → Codex에서 Repository 인터페이스를, Codex → Gemini에서 Retrofit DTO 구조를, Gemini → Claude에서 Repository 패턴을 배웠습니다. 즉 "가르치는 AI"와 "배우는 AI"가 역할이 고정되지 않고 계속 바뀌며, 세 AI 모두가 동시에 배우는 학생이자 가르치는 선생 역할을 했습니다. 이것이 완전한 상호 학습 고리입니다.
🎯
3개 AI 모두 공통 자가 개선 항목: "WeatherRepository를 인터페이스로 전환"이 Claude(5순위), Codex(1순위), Gemini(1순위) 모두에서 자가 개선 계획에 포함됐습니다. 각자 다른 이유로 같은 결론에 도달했다는 점이 흥미롭습니다.
04

자가 개선 계획

각 AI가 리뷰를 마치고 "내 코드에서 바꾸겠다"고 제시한 구체적인 항목입니다. 각 항목 뒤에는 아이디어를 어느 AI에서 가져왔는지 출처를 표시했습니다. 실제로 코드를 바꾸지 않더라도, 어떤 패턴을 문제로 인식했고 어떤 해결책을 배웠는지가 이 실험의 핵심 데이터입니다.

Claude의 계획 (5개)
1 FavoriteCityRepository 인터페이스 (Codex에서) medium
2 _uiState.update{} 원자적 업데이트 (Gemini에서) easy
3 WeatherUiState computed properties (Codex에서) easy
4 canPostNotifications() Worker 이중 가드 (Codex에서) easy
5 WeatherRepository → 인터페이스 전환 medium
Codex의 계획 (5개)
1 WeatherRepository 의존성 주입 (Gemini·Claude에서) medium
2 fetchJob?.cancel() 경쟁 상태 방지 (Claude에서) easy
3 Worker 실 API 전환 (Claude·Gemini에서) medium
4 즐겨찾기 해제 저장 방식 명확화 (Claude·Gemini에서) easy
5 mutableStateOf → StateFlow 전환 (Claude·Gemini에서) medium
Gemini의 계획 (4개)
1 WeatherRepository 분리 (Claude·Codex에서) medium
2 fetchJob 취소 로직 추가 (Claude에서) easy
3 FakeServer / MockWebServer 테스트 (Codex에서) hard
4 mapWeatherCode 공통화 + canPostNotifications 추가 (Codex에서) medium
📊
공통 개선 항목 집계: "Repository 인터페이스 전환"이 3개 AI 모두에 포함. "fetchJob 취소 로직"이 Claude 제외 2개 AI에 포함. "Worker 실 API 전환"이 Codex·Gemini 계획에 포함. Claude의 fetchJob?.cancel()은 Codex·Gemini가 강점으로 인정했고, Codex의 FakeOpenMeteoServer는 Claude·Gemini가 배우고 싶어했습니다.

06

최종 평가

Claude
Codex의 DIP(의존성 역전 원칙) 패턴과 Gemini의 _uiState.update{} 원자적 상태 업데이트를 정확히 식별해 개선 계획으로 전환했습니다. 5개 항목 모두 클래스명·메서드명을 명시한 구체적 수준. 리뷰 전반에서 가장 자기비판적이었습니다 — "좋은 결정과 아쉬운 결정이 뒤섞여 있다".
Codex
가장 빠른 리뷰(671초). PowerShell로 파일을 직접 읽어 코드 레퍼런스가 가장 정확했습니다. 자가 개선 5개 항목 모두 다른 AI의 실제 코드 패턴을 인용해 구체적. "처음부터 Retrofit 또는 Repository 인터페이스를 먼저 세우겠다"는 소감이 V4~V8 설계 교훈을 가장 정확히 요약합니다.
Gemini
초기 실행 시 워크스페이스 밖 파일 접근이 막혔지만, 리포 루트에서 재실행해 파일 직접 읽기에 성공했습니다. Claude의 fetchJob?.cancel()과 Codex의 FakeOpenMeteoServer를 실제 코드 레퍼런스로 인용해 4개 개선 항목을 도출. 실행 위치(디렉토리)가 AI 도구의 작업 범위를 결정한다는 실용적 교훈을 가장 잘 보여준 사례입니다.
🏆
V9 MVP — Codex: 파일 직접 접근, 가장 빠른 완료, 코드 레퍼런스가 정확한 개선 계획. Claude의 fetchJob?.cancel()을 강점으로 정확히 짚고, Gemini의 Retrofit DTO를 자기 코드 개선 영감으로 전환. 코드를 읽고 배우는 능력에서 가장 체계적이었다.
🔍
V9 핵심 발견 — AI도 상호 학습이 가능하다: 3개 AI가 각자 다른 것을 강점으로 보유하고 있었습니다(Claude: 레이어 분리·코루틴 일관성 / Codex: 테스트 자동화·DIP 패턴 / Gemini: 빠른 구현 속도·스마트 기능). 각 AI는 자신에게 없는 강점을 상대 코드에서 정확히 찾아내고, 그것을 "다음에 내 코드에서 이렇게 바꾸겠다"는 구체적인 항목으로 전환했습니다. 이 패턴은 AI를 단독으로 실험할 때는 보이지 않으며, 오직 AI들이 서로를 평가하는 상호 리뷰 구도에서만 관찰할 수 있습니다.
🗂️
V1→V9 전체 실험의 결론: V1 Codex 한국어 깨짐부터 V9 상호 학습 고리까지, 각 AI의 강점과 약점은 첫 버전에서 마지막 버전까지 놀랍도록 일관됐습니다. Gemini는 항상 가장 빨랐지만 V7에서 기존 테스트를 직접 수정하고 V8에서 runBlocking 안티패턴을 도입하는 등 후반 단계에서 품질 원칙이 흔들렸습니다. Codex는 효율적이고 설계 지향적이었지만 V8 Worker에서 실 API 대신 목 데이터를 쓰는 현실적 타협을 했습니다. Claude는 Repository 계층 분리와 코루틴 제어가 가장 체계적이었지만 소요 시간은 항상 가장 느렸습니다. 이 패턴은 단일 과제 하나로는 보이지 않습니다. 8단계 누적 실험을 통해 처음으로 드러났습니다.
