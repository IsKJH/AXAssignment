V7 — Room Database 영속성

즐겨찾기 도시를 앱 재시작 후에도 기억하도록 Room DB(안드로이드 공식 SQLite 래퍼 라이브러리)를 추가했습니다. DB를 추가하면 기존 단위 테스트가 "DAO(Data Access Object — DB 쿼리를 담당하는 인터페이스)를 모르기 때문에" 깨지는 문제가 생기는데, 이 테스트 호환성 문제를 세 AI가 완전히 다른 방식으로 해결했습니다: Claude(DAO nullable) / Codex(Repository 인터페이스·DIP) / Gemini(기존 테스트 직접 수정).

V5 연속성 — 왜 테스트 호환성이 핵심인가
V5에서 각 AI가 작성한 단위 테스트는 ViewModel을 생성자 인자 없이 인스턴스화합니다. V7에서 DAO를 ViewModel 생성자에 추가하면, 그 테스트들은 "DAO를 넘기지 않았다"는 이유로 컴파일 단계부터 실패합니다. 이것이 테스트 호환성 문제입니다. 새 기능을 추가하면서도 V5 테스트를 무수정으로 통과시키는 것이 이번 단계의 핵심 기술 과제였습니다.

03

테스트 호환 설계 전략

테스트 호환성(Test Compatibility)이란, 새 기능을 추가한 후에도 이전 단계에서 작성한 단위 테스트가 수정 없이 그대로 통과할 수 있는 상태를 말합니다. Room DAO를 ViewModel 생성자에 추가하면, V5 단계에서 작성된 테스트는 DAO를 인자로 넘기지 않기 때문에 컴파일 자체가 실패합니다. 세 AI가 이 문제를 각각 다른 방식으로 해결했습니다.

AI 전략 핵심 코드 장점 단점
Claude DAO nullable 파라미터
DAO가 null이면 DB 작업 생략		 class WeatherViewModel(private val dao: FavoriteCityDao? = null) 기존 테스트 코드를 한 줄도 건드리지 않아 테스트 호환성을 완벽히 유지합니다. 변경 범위가 최소화되어 도입 위험이 낮습니다. DAO가 null일 수 있다는 사실이 프로덕션 코드에 dao?.upsert() 형태로 남습니다. 실제로는 프로덕션에서 null이 들어올 일이 없는데도 코드가 방어적으로 작성되어, 타입 시스템이 "DAO가 없는 상태"를 정상 경로로 오인할 수 있습니다.
Codex Repository 인터페이스 + NoOp 구현체
테스트에선 아무것도 안 하는 NoOp, 프로덕션에선 Room 구현체 주입		 interface FavoriteCityRepository
object NoOpFavoriteCityRepository		 의존성 역전 원칙(DIP, Dependency Inversion Principle) — "상위 모듈(ViewModel)은 구체적인 구현(Room)에 의존하지 말고 추상(인터페이스)에 의존해야 한다"는 원칙 — 을 적용했습니다. ViewModel은 FavoriteCityRepository 인터페이스만 알고, 어떤 구현체가 주입되는지 관심이 없습니다. NoOpFavoriteCityRepository는 인터페이스를 구현하되 실제 저장은 아무것도 하지 않는 "빈 껍데기" 구현체로, 테스트에서 DB 없이 주입할 수 있습니다. Hilt·Koin 같은 DI 프레임워크를 도입하거나 구현체를 교체할 때 ViewModel 코드를 전혀 수정하지 않아도 됩니다. 파일 5개 추가로 코드량이 늘어납니다. 소규모 프로젝트에서는 과설계처럼 보일 수 있습니다.
Gemini 기존 테스트 직접 수정
DAO를 MockK로 mock해서 기존 테스트에 주입		 val mockDao = mockk<FavoriteCityDao>() 프로덕션 코드가 가장 단순하고 명확합니다. DAO가 직접 주입되므로 의도가 코드에 바로 드러납니다. 이 실험의 V5 단계 원칙인 "기존 테스트는 수정하지 말 것"을 직접 위반합니다. 테스트를 수정 가능한 산출물로 취급하면, 이후 V8·V9 단계에서도 기능 추가 때마다 테스트가 변경되어 "테스트가 설계 변경으로부터 코드를 보호한다"는 역할을 잃게 됩니다. 또한 DAO 시그니처가 바뀔 때마다 테스트도 함께 수정해야 하는 결합도 문제가 생깁니다.
🏆
Codex의 Repository 인터페이스 접근이 가장 설계 지향적입니다. DIP(Dependency Inversion Principle — 추상 인터페이스에 의존해 구현체를 교체 가능하게 하는 설계 원칙)를 적용해 ViewModel이 구체 클래스(Room DAO) 대신 추상(인터페이스)에 의존하도록 했습니다. 덕분에 테스트에서는 가짜 NoOp 구현체를, 실제 앱에서는 진짜 Room DB 구현체를 교체해 끼울 수 있습니다. 나중에 DataStore나 다른 저장소로 교체해도 ViewModel 코드를 건드리지 않아도 됩니다. 약점은 파일 수 증가로 인한 초기 코드량이지만, 프로젝트 규모가 커질수록 이 투자는 상쇄됩니다.
💡
Claude의 nullable DAO는 방어적 타협입니다. 기존 테스트를 수정하지 않으면서 가장 적은 코드 변경으로 목표를 달성한 실용적 선택입니다. 강점은 변경 범위가 최소화되고 테스트 호환성이 완벽하다는 점입니다. 약점은 DAO가 null일 수 있다는 사실이 프로덕션 코드에 영구적으로 남아, 실제로는 발생하지 않는 null 경로를 코드가 계속 인지하고 있어야 한다는 점입니다.
⚠️
Gemini의 기존 테스트 수정은 왜 문제인가. 이 실험은 V5 단계부터 "기존 테스트는 수정하지 말 것"을 명시적 원칙으로 설정했습니다. 이 원칙이 존재하는 이유는 단위 테스트가 코드 변경의 안전망이기 때문입니다. 테스트를 수정하면 그 안전망 자체가 흔들리고, 테스트가 "기능이 올바른지 확인"하는 역할 대신 "새 기능에 맞춰 계속 따라가야 하는 대상"으로 성격이 바뀝니다. Gemini는 프로덕션 코드를 가장 단순하게 작성했지만, 지시 이행 정확도와 장기적 설계 원칙 준수 측면에서 다른 두 AI와 차이가 있습니다.
