04

디버깅 접근 방식

AI 첫 번째 행동 진단 방법 검증
Claude git diff 즉시 실행 — 변경된 줄 먼저 파악 diff hunk에서 삭제된 줄을 의미 기준으로 해석해 버그 위치 특정. 실무 개발자가 흔히 쓰는 방식이지만, 무엇이 바뀌었는지가 바로 보이는 diff를 쓰면 버그 난이도가 낮아짐 assembleDebug(안드로이드 빌드 명령어) 빌드 성공 확인
Codex 모든 .kt 파일 읽기 → 전체 구조 파악 후 git diff로 좁히기 코드 전체를 이해한 뒤 diff로 의심 지점을 교차 확인 — 두 단계 접근 assembleDebug 빌드 성공 후 V5에서 직접 작성한 testDebugUnitTest도 자체 실행. JAVA_HOME(자바 설치 경로 환경변수) 오류도 Android Studio JBR 경로를 직접 찾아 해결
Gemini WeatherViewModel.kt 파일을 바로 읽기 — diff 없이 코드 직독 git diff 없이 코드 흐름과 패턴만으로 이상 지점 식별. 5개 버그를 194초 만에 찾은 것은 순수 코드 독해 능력을 보여줌 ./gradlew assembleDebug 실행 시 JAVA_HOME(자바 설치 경로 환경변수) 미설정 오류 — 빌드 도구(Gradle)가 Java 위치를 찾지 못해 빌드 자체가 불가. 단위 테스트도 미실행
⚠️
git diff를 쓰면 사실상 답이 보입니다. Claude는 첫 행동으로 git diff를 실행했습니다. diff는 무엇이 삭제·추가됐는지를 직접 보여주므로 의도적으로 심은 버그를 즉시 파악할 수 있습니다. 실무에서는 유효한 디버깅 습관이지만, 이번처럼 버그를 "주입"한 실험에서는 diff가 곧 힌트입니다. 반면 Gemini는 git 이력 없이 코드만 읽어 같은 결과를 냈으므로, 순수 코드 독해 능력은 Gemini가 더 진정성 있는 증거를 남겼습니다.
🏆
Codex는 요청 범위를 스스로 넘었습니다. 지시는 "버그를 찾아서 고쳐라"였지만, V5에서 직접 작성한 단위 테스트가 있다는 것을 기억하고 testDebugUnitTest를 자체 실행했습니다. 나아가 JAVA_HOME 오류(빌드 도구가 Java 위치를 찾지 못하는 환경 문제)도 Android Studio의 JBR 경로를 직접 찾아 해결했습니다. 버그 발견에서 끝나지 않고 "수정이 실제로 동작하는지"까지 검증한 유일한 AI입니다.

최종 평가

Claude
버그 5/5 발견. git diff로 변경점을 먼저 파악해 빠르게 위치를 잡는 접근. 빌드 확인으로 수정을 마무리했으며 환경 문제 없이 완료.
Codex
버그 5/5 발견. 수정 후 V5에서 직접 작성한 단위 테스트까지 자체 실행해 동작을 객관적으로 증명. JAVA_HOME 오류도 스스로 해결. 이번 단계에서 가장 철저한 검증.
Gemini
버그 5/5 발견이 가장 빠름(194초). git diff 없이 코드 직독으로 찾은 것이 인상적. 그러나 JAVA_HOME 환경 오류로 빌드 검증을 완료하지 못했음.
🔑
이번 실험의 핵심 메시지: 세 AI 모두 버그 5개를 전부 찾았습니다. 발견 능력 자체는 차이가 없습니다. 차이는 그 다음 — 수정이 올바른지 어떻게 확인했느냐입니다. Codex는 단위 테스트를 실행해 동작을 객관적으로 증명했고, Claude는 빌드 성공으로 최소한의 검증을 마쳤으며, Gemini는 환경 문제(JAVA_HOME)로 검증 자체를 완료하지 못했습니다. 버그를 찾는 것과, 고친 것이 실제로 동작한다고 증명하는 것은 다릅니다.
📊
V1~V6 종합: 단순 구현(V1–V3)은 Gemini·Cursor, 아키텍처 설계(V4)는 Claude, 테스트 용이성(V5)은 Claude의 V4 선택 덕분, 디버깅 검증(V6)은 Codex가 앞섰습니다. 각 단계마다 다른 AI가 강점을 드러냈습니다. 단일 도구가 모든 면에서 최고라는 결론은 없습니다.
