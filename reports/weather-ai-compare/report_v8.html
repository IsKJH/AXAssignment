V8 — WorkManager 백그라운드 알림

WorkManager(앱이 꺼진 상태에서도 안드로이드 OS가 주기적으로 실행해주는 백그라운드 작업 스케줄러)로 즐겨찾기 도시 날씨 알림을 15분마다 발송하고, 실기기에서 ADB(안드로이드 기기 연결 및 명령 실행 도구) 설치 → 강제 트리거 → 스크린샷까지 직접 검증했습니다. 핵심 교훈: Gemini는 assembleDebug(안드로이드 빌드 명령어)를 통과했지만 실기기에서 즉시 크래시했고, 4개 버그를 스스로 발견·수정한 뒤에야 알림을 수신했습니다. 빌드 성공이 실동작을 보장하지 않는다는 사실이 이번 실험의 핵심 발견입니다.

실험 설계
WorkManager는 앱이 포그라운드에 없거나 기기가 절전 상태여도 OS가 대신 Worker 코드를 깨워 실행해주는 안드로이드 공식 백그라운드 작업 스케줄러입니다. V7에서 Room DB로 저장한 즐겨찾기 도시를 Worker가 읽어 15분마다 알림을 발송합니다. Android 8.0+ 알림 채널, Android 13+ 런타임 권한, 앱 시작 시 스케줄 등록이 필수 조건입니다.
핵심 관찰 포인트
이번에는 V7과 달리 세 AI 모두 기존 테스트를 수정하지 않았습니다. 대신 Worker 내부의 코루틴 패턴(올바른 suspend fun doWork() vs. 잘못된 runBlocking 중첩)과 권한 체크 위치에서 설계 철학 차이가 드러났습니다. 그리고 빌드 결과만으로는 이 차이를 발견할 수 없다는 점이 중요합니다.
V7 연속성
Room에 저장된 즐겨찾기 도시 인덱스를 Worker가 읽어서 알림 내용을 결정합니다. V7의 DAO 접근 방식(Claude nullable, Codex Repository 인터페이스, Gemini direct 접근)이 Worker 구현에도 그대로 이어집니다.

⚠️
Gemini — runBlocking 안티패턴 (그리고 왜 위험한가): CoroutineWorker는 이미 코루틴 위에서 동작하도록 설계된 클래스입니다. doWork()를 suspend fun으로 선언하면 코루틴 런타임이 스레드를 막지 않고 비동기로 처리합니다. 그런데 Gemini는 override fun doWork()(suspend 없음)로 선언하고 내부에서 runBlocking { ... }으로 감쌌습니다. 비유하자면: 이미 자동으로 달리는 에스컬레이터(CoroutineWorker) 위에서 직접 달리겠다며 에스컬레이터를 통째로 세우는(runBlocking) 것입니다. runBlocking은 코루틴을 현재 스레드에서 완전히 끝날 때까지 블로킹합니다 — 즉, 코루틴이 비동기로 처리되는 것을 막고 스레드가 결과를 기다리며 멈추게 합니다. 결과적으로 WorkManager 스레드 풀의 스레드 하나가 작업이 끝날 때까지 잠겨버립니다. 빌드는 통과하지만 실기기에서 ClassNotFoundException으로 즉시 크래시했고, 배터리 효율·Doze 모드 동작에서도 예기치 않은 결과를 낳을 수 있습니다.
🔒
Codex — 이중 권한 방어: Android 13+ 권한을 MainActivity에서 요청할 뿐 아니라, Worker 내부에서도 canPostNotifications()를 통해 발송 직전 다시 확인합니다. 권한이 사후에 취소되더라도 알림 발송 예외를 방지하는 가장 방어적인 접근입니다.
🗂️
Codex — 관심사 분리 일관성: V7에서 FavoriteCityRepository 인터페이스로 분리했던 것처럼 V8에서도 WeatherNotificationScheduler를 별도 클래스로 분리했습니다. Worker 등록 로직이 Application 클래스에 섞이지 않아 테스트와 교체가 쉽습니다. 반면 Claude·Gemini는 Application 클래스에 직접 인라인했습니다.
❓
Codex — 왜 V8에서 목 데이터를 썼나: Codex는 V4부터 Open-Meteo 실 API를 사용해왔습니다. 그런데 V8 Worker에서는 mockConditionForCity(cityIndex)로 하드코딩된 날씨 상태("맑음", "구름많음" 등)를 반환했습니다. 이유는 Worker의 책임 범위를 "알림 발송"으로 한정하고 네트워크 지연 실패를 Worker 재시도 루프와 분리하려는 설계 판단으로 보입니다. 알림 신뢰성(네트워크 없어도 항상 발송)을 API 정확성보다 우선했습니다. Claude·Gemini는 반대로 실 API를 Worker 안에서 직접 호출하여 실시간 날씨를 제공하되, 실패 시 Result.retry()로 대응했습니다.

🔧
Gemini — 크래시 → 자가수정 4단계 과정:

1단계 · 빌드 성공 착각: Gemini의 초기 구현은 assembleDebug를 통과했습니다. 컴파일러가 보기에 문법 오류가 없었기 때문입니다. 그러나 이 시점에 이미 4개의 런타임 시한폭탄이 심어져 있었습니다.

2단계 · 실기기 즉시 크래시: adb install 후 앱을 실행하자마자 ClassNotFoundException이 발생했습니다. 안드로이드 런타임이 Worker 클래스를 찾지 못한 것입니다. assembleDebug는 이 문제를 전혀 잡지 못했습니다.

3단계 · 원인 분석 및 수정: Gemini가 직접 로그를 분석해 4개 원인을 특정했습니다. ① kotlin-android 플러그인이 build.gradle에 누락되어 코틀린 클래스가 제대로 컴파일되지 않음 ② doWork()의 suspend 키워드 누락으로 CoroutineWorker 오버라이드 계약 위반 ③ AGP 9.2.1 등 미출시 실험적 버전 사용으로 런타임 불안정 ④ gradle.properties의 android.useAndroidX=true 누락으로 AndroidX 라이브러리 클래스 로딩 실패. 이 중 ①②④는 assembleDebug가 통과시킨 런타임 버그입니다.

4단계 · 재시도 성공: 4개를 모두 수정한 뒤 재빌드 → 재설치 → WorkManager 강제 트리거 → "GEMINI 서울 날씨 알림 · 대체로 맑음" 알림 수신 성공. 하지만 이 과정이 전체 검증 시간을 2,823초(다른 AI의 약 2~2.7배)로 만들었습니다.
📌
assembleDebug 통과 ≠ 실기기 실동작 보장 — 이번 실험의 핵심 교훈: assembleDebug는 소스 코드를 DEX 바이트코드로 변환할 때 발생하는 컴파일 오류만 잡습니다. 플러그인 누락, 라이브러리 버전 충돌, 클래스 로딩 실패, API 레벨 불일치 같은 런타임 문제는 기기에서 실제로 실행해봐야만 발견됩니다. Gemini의 V8 사례는 "빌드 성공 = 완성"이라는 착각이 얼마나 위험한지를 보여줍니다. ADB 설치 → WorkManager 강제 트리거 → 스크린샷 확인이라는 실기기 검증 단계가 이 실험에 필수였던 이유입니다.

최종 평가

Claude
실 API 연동으로 가장 완성도 높은 알림 내용, 정석 suspend 패턴, 깔끔한 최소 구현. 별도 클래스 없이 Application에 직접 통합.
Codex
가장 빠르고 토큰 효율 최고. Worker 내 이중 권한 체크, BigTextStyle 알림, 커스텀 아이콘, Scheduler 클래스 분리까지 — 세부 완성도가 가장 높음.
Gemini
실 API 연동, 빌드 성공. 그러나 runBlocking 안티패턴이 명확한 기술 부채. 도구 오류로 시간도 가장 길었음.
🏆
V8 MVP — Codex: 속도(410초)·토큰 효율(74,945)·구현 완성도 세 항목 모두에서 앞섰습니다. CoroutineWorker의 suspend fun doWork() 정석 패턴을 따르면서, 권한 이중 방어(canPostNotifications()를 Worker 내부에서 한 번 더 확인)와 알림 UX(BigTextStyle + 커스텀 벡터 아이콘)는 Claude·Gemini에 없는 추가 배려입니다. 목 데이터 선택은 네트워크 실패와 알림 발송 실패를 분리해 알림 신뢰성을 우선한 실용적 판단입니다.
📐
V7 → V8 일관성: Codex는 V7의 FavoriteCityRepository 인터페이스 패턴을 V8의 WeatherNotificationScheduler 분리까지 이어갔습니다. Claude는 V7의 nullable 파라미터 실용주의를 V8에서도 유지했고, Gemini는 V7에서 테스트를 직접 수정한 것처럼 V8에서 코루틴 패턴을 어겼습니다. 두 버전에 걸쳐 각 AI의 설계 철학이 일관되게 나타났습니다. 한 번의 설계 판단이 버전을 넘어 반복된다는 점에서, 초기 코드 리뷰에서 패턴을 잡는 것이 중요함을 보여줍니다.
