01

실험 개요

V4에서 실제 API를 붙인 앱에 단위 테스트를 추가하는 작업입니다. 프로덕션 코드를 수정하지 말고, 5가지 시나리오(초기 로딩·도시 선택·즐겨찾기·에러·성공)를 JVM 유닛 테스트(실기기 없이 컴퓨터에서 실행되는 단위 테스트)로 커버하라는 동일한 요구사항을 세 AI에게 주었습니다. 단위 테스트의 핵심은 "실제 네트워크 없이 ViewModel 로직만 검증하기"이며, 이를 위해 외부 의존성을 가짜로 교체하는 것이 모킹(mocking — 실제 네트워크 없이 테스트하기 위해 가짜 응답을 주입하는 기법)입니다.

비교 초점
각 AI가 ViewModel의 Repository/Network 의존성을 테스트 환경에서 어떻게 격리하는지를 봅니다. 세 프로젝트의 V4 아키텍처가 서로 달랐기 때문에, 같은 요구사항에도 불구하고 모킹 진입점이 달라졌고 그 결과 전략도 달라졌습니다. "어디를 교체하면 테스트가 가능한가"라는 질문의 답이 V4 설계에서 이미 결정돼 있었습니다.
핵심 발견
Claude는 MockK로 싱글톤 Repository 자체를 교체했고(객체 레벨), Codex는 MockK 없이 JVM의 HTTP URL 처리 메커니즘을 가로채 가짜 응답을 주입했으며(네트워크 레벨), Gemini는 MockK + Turbine으로 Retrofit 서비스 생성 시점을 교체하고 Flow 방출 순서까지 정밀 검증했습니다(서비스 팩토리 레벨). 같은 "테스트"라도 어느 레이어를 잘라내느냐가 완전히 달랐습니다.
공통 결과
세 AI 모두 5개 테스트를 최종 통과시켰습니다. Codex만 1차 실행에서 3개가 실패했습니다. 단순한 실수가 아니라 디스패처 선택이 맞지 않아 코루틴이 시간 초과된 구조적 문제였고, Codex는 원인을 스스로 진단해 수정 후 전부 통과시켰습니다.

모킹 전략 비교

같은 요구사항에서 AI마다 완전히 다른 테스트 격리 전략을 선택했습니다. 각 전략은 단순한 취향 차이가 아니라 V4 아키텍처가 허용한 "교체 가능한 지점"이 어디였는지를 그대로 반영합니다.

AI격리 전략의존성 접근Dispatcher 선택특이점
Claude mockkObject(WeatherRepository)
— object 싱글톤을 MockK로 직접 교체.

장점: 가장 단순하고 직관적. ViewModel이 Repository를 호출할 때 가짜 구현이 대신 응답하므로 ViewModel 로직만 순수하게 검증 가능.
단점: mockkObject는 JVM 전역 상태를 변경하므로 테스트 후 반드시 unmockkObject로 원상복구해야 함. 복구를 빠뜨리면 다른 테스트에 영향을 줄 수 있음.		 V4에서 WeatherRepository를 독립적인 object로 분리한 덕분에 ViewModel 코드를 한 줄도 건드리지 않고 테스트 가능. Clean Architecture가 테스트 용이성으로 직접 이어진 사례. StandardTestDispatcher + Dispatchers.setMain + advanceUntilIdle() 프로덕션 코드 무수정. 테스트 코드 140줄로 세 AI 중 최소.
Codex URLStreamHandlerFactory 직접 구현
— https:// 프로토콜을 JVM 레벨에서 가로채 가짜 HTTP 응답을 주입.

이게 무엇인가: Java/JVM에서 URL.openConnection()이 실제 소켓을 열기 전에 어떤 팩토리를 거쳐 연결 객체를 만들지를 결정합니다. URLStreamHandlerFactory를 교체하면 JVM 전체에서 실제 네트워크 없이 원하는 응답을 리턴할 수 있습니다. MockK 같은 라이브러리 없이 JVM 표준 API만으로 네트워크를 가로채는 저수준 기법입니다.

장점: 실제 HTTP 스택(OkHttp → URLConnection)에 가장 가까운 레이어를 가로채기 때문에 현실과의 괴리가 적음. 라이브러리 의존성 추가 없음.
단점: 구현 코드가 복잡하고 길어짐(245줄). URLStreamHandlerFactory는 JVM 당 한 번만 등록 가능해 병렬 테스트 시 충돌 위험. org.json 라이브러리를 별도 추가해야 함(JVM 환경에서 Android의 JSONObject 클래스를 사용할 수 없으므로).		 V4에서 ViewModel 내부에 Repository/네트워크 로직이 함께 있어 MockK로 교체할 명확한 진입점이 없었음. ViewModel 자체를 건드리지 않고 테스트하려면 더 아래인 네트워크 레이어까지 내려가야 했고, 결국 URL 처리 레이어에서 우회하는 방식을 선택. UnconfinedTestDispatcher (1차 시도에서 StandardTestDispatcher를 사용했다가 3개 실패 → 자가 수정) MockK 미사용. org.json 라이브러리 추가 필요. 가장 많은 테스트 코드(245줄).
Gemini mockkObject(WeatherApiService.Companion)
— Retrofit 서비스를 생성하는 팩토리 메서드(Companion.create())를 MockK로 교체.

장점: Retrofit 레이어를 통째로 잘라내므로 HTTP 직렬화/역직렬화 없이 깔끔하게 격리. Turbine을 통해 Flow가 어떤 순서로 값을 방출하는지 시퀀스 단위로 검증 가능(단순히 최종 상태만 보는 것보다 정확).
단점: Turbine이라는 추가 라이브러리 의존성이 생기고, Companion 객체 모킹은 MockK 특유의 문법이라 처음 보면 다소 낯설 수 있음.		 V4에서 Retrofit WeatherApiService를 사용하므로 서비스 인스턴스 생성 시점(Companion.create())을 MockK로 교체. Turbine은 코루틴 Flow(상태 변화를 구독할 수 있는 코루틴 기반 데이터 흐름)의 방출 순서를 awaitItem()으로 순서대로 단언하는 전용 라이브러리로, 비동기 상태 변화 검증을 직관적으로 표현. StandardTestDispatcher + MainDispatcherRule 프롬프트에서 "optional"로 제시한 Turbine을 유일하게 채택. Flow 방출 시퀀스를 단계별로 검증하는 표현력 높은 테스트 작성.

Codex의 V4에서 ViewModel 안에 로직을 통합한 것이 V5에서 MockK 진입점을 없애버렸고, 결국 JVM URL 스택을 직접 가로채는 245줄짜리 우회로가 필요해졌습니다. 105줄의 차이는 기능이 아니라 이전 설계 결정의 비용입니다. 좋은 설계는 지금 당장이 아니라 다음 단계에서 그 가치가 드러납니다.

Codex 자가수정 과정 — 실패에서 통과까지

1단계 · 실패 발생: 1차 실행에서 selectCity, toggleFavorite, networkFailure 3개 테스트가 TimeoutCancellationException으로 실패했습니다. 에러 메시지만 보면 "시간이 초과됐다"는 것 외에 원인이 명확하지 않습니다.

2단계 · 원인 진단: Codex는 로그를 분석해 근본 원인을 정확히 짚었습니다. StandardTestDispatcher는 코루틴을 즉시 실행하지 않고 큐에 넣어 두는 방식입니다. 테스트 코드에서 awaitLoaded()라는 헬퍼 함수가 ViewModel이 로딩을 완료하기를 기다리고 있었는데, StandardTestDispatcher 특성상 실제 코루틴 작업이 아직 실행 전 대기 중인 상태였습니다. 결과적으로 awaitLoaded()는 영원히 오지 않을 완료 신호를 기다리다 타임아웃이 발생한 것입니다. "닭이 먼저냐 달걀이 먼저냐"와 같은 데드락 상황이었습니다.

3단계 · 자가 수정: Codex는 UnconfinedTestDispatcher로 교체했습니다. 이 디스패처는 코루틴을 큐에 넣지 않고 즉시 실행하므로 awaitLoaded() 호출 시점에 이미 코루틴이 완료돼 있습니다. 추가로 Dispatchers.Default.limitedParallelism(1)을 조합해 병렬 실행으로 인한 경쟁 상태도 차단했습니다. 수정 결과 5/5 전부 통과.

주목할 점: 단순히 "실패했으니 다시 시도"가 아니라, 디스패처 동작 원리를 이해한 정확한 진단과 교과서적인 수정이었습니다. 이 자가수정 과정에서도 총 소요 시간은 256초로 Claude(428초)보다 빨랐습니다.

핵심 인사이트

🏗️
V4 아키텍처 결정이 V5 테스트 복잡도를 직접 결정했습니다. 테스트 코드량 차이(140줄 vs 245줄)는 단순히 "Codex가 더 많이 썼다"는 이야기가 아닙니다. Claude의 V4에서 WeatherRepository를 독립 객체로 분리했기 때문에 V5에서 MockK 한 줄로 교체 가능했습니다. Codex의 V4에서 ViewModel에 로직이 통합돼 있었기 때문에 교체 지점이 없어졌고, 105줄 분량의 JVM URL 인터셉터 인프라를 직접 구현해야 했습니다. 추가된 코드는 기능이 아니라 V4의 아키텍처 결정이 남긴 부채입니다. 단위 테스트 추가가 힘들다면 그것은 "테스트 작성이 어렵다"가 아니라 "설계가 테스트를 막고 있다"는 신호입니다.

Codex는 혼자서 실패를 진단하고 고쳤습니다. 1차 실행에서 3개 테스트가 TimeoutCancellationException으로 실패했을 때, Codex는 단순히 재시도하거나 테스트를 제거하지 않았습니다. StandardTestDispatcher의 지연 실행 특성이 awaitLoaded()와 데드락을 만든다는 것을 정확히 진단하고, UnconfinedTestDispatcher로 교체해 즉시 실행되도록 수정했습니다. 자가 진단 → 원인 특정 → 교과서적 수정까지 이 과정을 스스로 완료했고, 그 결과 최종 256초로 Claude(428초)보다 빠르게 마쳤습니다.
📦
Gemini만 Turbine을 선택했습니다 — 그리고 그것은 의미 있는 선택이었습니다. 프롬프트에서 "optional"로 제시했는데 Gemini만 채택했습니다. 단순히 라이브러리를 더 쓴 것이 아니라, Turbine을 사용하면 Flow가 "isLoading=true → isLoading=false → weatherData 설정"처럼 방출하는 값의 순서 자체를 awaitItem()으로 단계별로 단언할 수 있습니다. Claude와 Codex의 테스트가 최종 상태만 확인하는 반면, Gemini의 테스트는 상태 전이 시퀀스를 검증합니다. 어느 쪽이 낫다기보다, Gemini가 Flow 기반 상태 관리를 가장 정밀하게 검증했다고 볼 수 있습니다.
